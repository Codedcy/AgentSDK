# Agent SDK v0.1 Release Convergence Design

Status: approved in conversation on 2026-07-17; detailed R0-R5 implementation
plans prepared under `docs/superpowers/plans/`.

## 1. Purpose

The first release prioritizes a usable end-to-end Agent SDK over production-grade
hardening. Version `0.1.0` must provide the basic behavior of every core capability,
remain observable, and recover from the last completed safe boundary. It does not
promise exactly-once external effects, multi-worker recovery, or exhaustive
compatibility with arbitrary persisted or third-party inputs.

This document changes release ordering, not the long-term product direction. The
existing high-level and feature designs remain the long-term target. Requirements
outside this document move to the post-v0.1 hardening backlog unless they are needed
to prevent database loss, workspace escape, credential disclosure, or failure of the
v0.1 acceptance scenario.

## 2. Existing Baseline

The implementation already provides:

- an asyncio-first Agent Loop backed only by LiteLLM;
- SQLite by default, with explicit Memory construction for tests and transient use;
- Session and Run persistence, events, idempotency, leases, interruption records,
  reconciliation, and conservative recovery;
- user Tool registration, one permission flow, MCP Client integration, and Skills;
- a sequential Agent-only Workflow whose later nodes may be Child Runs;
- manual L0 Context Views and forced L3/L4 Context Capsules;
- Run timelines, execution trees, cursor queries and subscriptions;
- custom Evaluators, exact-output evaluation, success rate, Tool failure count, and
  Tool failure rate;
- a public-API reference CLI and an integrated M01 acceptance scenario.

The current code checkpoint is commit `0a83e46`. Its focused migration gates are
green, while the complete storage/full-project/build gates after that final small
fix remain pending. Those complete gates move to the v0.1 release-candidate gate.
M02-T003 Artifact publish/cleanup Phases B-D and M02-T004 advanced controls no
longer block v0.1.

## 3. v0.1 Product Contract

### 3.1 Stable functional surface

Version `0.1.0` exposes:

- Agent definition and Run execution with default and application-supplied system
  prompts;
- SQLite Session history, close/delete semantics, and safe-boundary recovery;
- user Tool registration and built-in `read`, `write`, and `bash` Tools;
- workspace-scoped `allow`, `deny`, and `ask` authorization;
- MCP Tools and progressively loaded Skills;
- Workflow definitions from YAML/JSON text or Python models, including sequential
  Agent nodes, conditions, and bounded loops;
- automatic L0-L4 Context planning before every model call;
- parent-created Child Runs, durable parent/child messaging, waiting, listing, and
  progress inspection;
- live and historical Trace queries;
- deterministic, evidence-linked basic evaluation and attribution;
- a buildable and installable wheel plus one complete reference application.

The API is stable within the v0.1 minor line, but reliability is limited by the
explicit boundaries in this document.

### 3.2 Explicit non-goals

Version `0.1.0` does not require:

- exactly-once Tool, MCP, filesystem, shell, or model side effects;
- simultaneous recovery by multiple SDK instances or distributed workers;
- cryptographic certification of every historical event relation;
- support for arbitrary SQLite DDL normalization or arbitrary migration backends;
- a filesystem Artifact generation/claim/cleanup state machine;
- Workflow parallel, foreach, subworkflow, quality-gate, compensation, general
  retry graph, or arbitrary expression execution;
- a fair or distributed Child scheduler, detach, dynamic budget transfer, or
  automatic deadlock resolution;
- vector retrieval, embeddings, or semantic search;
- causal attribution, LLM-as-judge attribution, statistical confidence modeling,
  or automatic optimization;
- a sync facade, force-delete recovery coordinator, hosted control service,
  dashboard, or product UI.

## 4. Release Acceptance Scenario

The installed wheel must execute one deterministic scenario through public APIs:

1. Create a SQLite Session with a configured workspace.
2. Define an Agent with a configurable system prompt.
3. Start a Run and automatically build a Context View before each model request.
4. Cross configured Context watermarks so L1, L2, L3, and L4 are each exercised
   and recorded without deleting the original Session ledger.
5. Invoke one application Tool, one built-in workspace Tool, and one MCP Tool
   through authorization.
6. Load one Skill and include its activated content in a Prompt Manifest.
7. Produce a Workflow candidate containing a condition and a bounded loop.
8. Validate the candidate, require application confirmation, and execute it.
9. Let the parent Agent invoke `spawn_agent`, exchange messages with the Child,
   inspect progress, wait for the result, and consume the Child output.
10. Subscribe to Trace events while work is active and query the complete timeline
    after completion.
11. Evaluate the result and return a basic attribution summary containing failure
    stage when applicable and the contributing Model, Tool, Context, Workflow, and
    Child records.
12. Terminate and reopen at selected safe boundaries, prove completed work is not
    repeated, and prove an unknown in-flight Model/Tool outcome becomes
    `interrupted` until the application chooses retry or abort.
13. Close and delete the Session, proving SDK-owned history is removed while
    workspace files remain.

No feature is considered part of v0.1 until it participates in this scenario.

## 5. Workflow v0.1

### 5.1 Node model

The v0.1 Workflow language contains:

- `agent`: execute a normal Run;
- `condition`: choose one branch using a restricted expression;
- `loop`: repeat a nested body until a restricted condition is true or
  `max_iterations` is reached.

Agent nodes continue to call registered Tools through the ordinary Agent Loop.
A separate Workflow Tool node is unnecessary for v0.1.

### 5.2 Restricted expressions

The Definition adds an immutable JSON-compatible `inputs` mapping. Conditions read
only those inputs and completed node outputs. Supported
operators are `eq`, `ne`, `gt`, `gte`, `lt`, `lte`, `contains`, and `exists`.
Paths are dotted object paths with array indexes. Expressions cannot call
functions, access Python attributes, perform I/O, or mutate state.

Every loop declares `max_iterations` between 1 and a configurable SDK maximum.
Reaching the maximum without satisfying the exit condition produces a visible
Workflow failure. Condition/loop nesting is statically bounded by a configurable
compiler maximum. There is no unbounded loop.

### 5.3 Persistence and recovery

The compiler produces immutable instructions with stable ids and a definition
hash. The Workflow snapshot persists the program counter, chosen condition branch,
loop iteration counters, completed Agent node results, and failure state.
Condition and loop instructions are deterministic and may be recomputed after
restart. Completed Agent Runs are never repeated. An Agent node with an unknown
in-flight result leaves the Workflow active and the Run `interrupted` until the
application resolves it.

Model-generated YAML is only a candidate. The application displays or otherwise
reviews it and explicitly calls the public start method after validation. Durable
proposal editing and approval state are post-v0.1 work.

## 6. Automatic Context L0-L4

Context planning runs before every model request through one Runtime middleware.
It records the selected level, source ids, token estimates, applied transformations,
usage, duration, fallback, and resulting Context View revision.

- `L0 Full`: retain all selected source text.
- `L1 Reference`: deduplicate repeated content and replace large Tool/MCP output
  with a bounded preview plus durable event references.
- `L2 Selective`: retain system instructions, current input, protected/pinned facts,
  unresolved requests, active Workflow/Child state, and a recent window; replace
  older completed progress and Tool detail with structured outcomes and references.
- `L3 Summary`: use LiteLLM structured output to summarize a closed older history
  segment into a validated Context Capsule while retaining recent protected items.
- `L4 Rebase`: merge earlier Capsules with the active objective, constraints,
  unresolved state, current Workflow/Child state, and recent messages into one new
  baseline plus source references.

Watermarks choose and apply a level automatically. The application may configure
thresholds, force a level, or disable lossy levels. L1/L2 are deterministic and
make no model call. L3/L4 validation or model failure falls back to L2 and emits
Trace evidence instead of failing the main Run. Original Session events remain
unchanged and retrievable. v0.1 retrieval is by explicit source references and
metadata; it does not include vector search.

## 7. Child Runs and Communication

The SDK registers four built-in control Tools for an Agent that has the matching
capability:

- `spawn_agent(task_envelope) -> child_run_id`;
- `send_message(target_run_id, message) -> message_id`;
- `wait_child(child_run_id, timeout_seconds) -> child_result_or_pending`;
- `list_children() -> child_progress[]`.

A Child is a normal Run with a durable parent relationship. Its initial Context
contains the TaskEnvelope, explicitly referenced evidence, permitted Skills, and
its own subsequent history. It does not inherit the complete parent conversation.

The effective Child Tool set and workspace roots are the intersection of parent
capabilities, TaskEnvelope limits, and Child Agent limits. An empty intersection
is allowed and means the Child has no Tool/workspace capability.

Messages are immutable, Session-owned records with sender, recipient, sequence,
created time, and content. Delivery is at least once. Before a model step, the
recipient injects unseen message ids and records the consumed ids in its Context
View so repeated reads do not duplicate model input. Both parent-to-child and
child-to-parent communication use the same mailbox. A sender may address only its
direct parent or direct children in the same Session; unrelated Runs are rejected
before any message write.

v0.1 enforces configurable maximum depth, children per parent, total children per
Session, and concurrent Child Runs. It uses a simple bounded semaphore, not a fair
or distributed scheduler. Parent cancellation/detach semantics are deferred; a
Session cannot be normally closed while a Child is active.

## 8. Trace and Basic Attribution

### 8.1 Trace facts

The event log remains the source of truth. Public Trace queries expose Run/Step,
Context, Model, Tool, Workflow/node, Child/message, permission, evaluation, and
recovery stages. Every applicable stage records:

- stable entity and parent ids;
- start/end time and duration;
- running, waiting, completed, failed, denied, timed-out, or interrupted status;
- model/tool/workflow/context revision identity;
- token usage and provider-reported cost when available;
- sanitized error code, failure stage, and retryability;
- input/output/evidence references rather than secret-bearing raw diagnostics.

The application decides when and how to display these records.

### 8.2 Attribution rules

`AttributionSummary` is deterministic and evidence-linked:

- failure is assigned to the first terminal failing stage in the execution path;
- Models, Tools, Context revisions, Workflow nodes, and Child Runs whose outputs
  were included in a later model request or terminal result are listed as
  contributors;
- a Tool result is `consumed` when its result id appears in a later Context View,
  otherwise `unused`;
- a Child result is consumed when its result/message id appears in the parent
  Context View;
- evaluation verdicts and evidence ids are attached to the terminal Run;
- improvement hints are fixed rules such as repeated Tool failure, unused Tool
  output, Context fallback, loop-limit exhaustion, or Child failure.

These records describe contribution and correlation, not causal proof. Cross-Run
statistics beyond existing success and Tool failure metrics are post-v0.1.

## 9. Recovery and Error Contract

SQLite is the default recoverable Store. A safe boundary exists after a committed
model outcome, Tool outcome, Workflow instruction, Context View, Child message, or
Child terminal result. Reopen continues from the latest boundary.

If the process stops after external work may have started but before its outcome is
committed, the owning Run becomes `interrupted`. The SDK performs no automatic
external replay. The application chooses:

- `retry`: execute the interrupted operation again and record the decision; or
- `abort`: fail the Run with an application-resolution reason.

The contract does not claim the first attempt did or did not execute. Recovery is
single-SDK/single-process at a time for v0.1. Concurrent worker recovery is outside
the supported contract even if existing internals handle some cases.

## 10. Delivery Slices

### R0 — Scope reset and release harness

- Freeze M02-T003 at `0a83e46`; defer Artifact Phases B-D and advanced M02-T004.
- Replace the milestone order for v0.1 with the slices below.
- Add one public-API v0.1 acceptance test and mark initially missing behaviors.
- Keep the old roadmap as the post-v0.1 hardening source rather than deleting it.

### R1 — Built-in Tools and basic policy

- Add `read`, `write`, and `bash`.
- Enforce canonical workspace containment and `allow/deny/ask`.
- Reuse the existing Tool registry, Tool results, permission bridge, and Trace.

### R2 — Workflow condition and bounded loop

- Extend the Definition, IR/compiler, snapshots, executor, and public result.
- Persist program counter, branch, and loop counters.
- Extend the acceptance scenario through a conditional bounded loop.

### R3 — Automatic L0-L4 Context

- Add deterministic L1/L2 strategies and real L4 rebase.
- Add the Runtime Context middleware before every model request.
- Persist fallback and source evidence without requiring File Artifact lifecycle.

### R4 — Child control Tools and mailbox

- Add spawn/message/wait/list Tools, enforced capability intersection, limits, and
  durable at-least-once mailbox consumption.
- Extend Trace and the acceptance scenario through a parent/child exchange.

### R5 — Attribution and release

- Add normalized stage facts and deterministic AttributionSummary.
- Complete the installed-wheel reference example and documentation.
- Run the release-candidate verification gate and publish version `0.1.0`.

Slices are sequential because each extends the same acceptance scenario. A slice
may be reviewed independently, but later work does not wait for repeated
whole-project reviews of the preceding slice. R0-R5 receive separate executable
implementation plans so each plan remains small enough to test and resume.

## 11. Verification and Review Policy

During implementation:

- use RED/GREEN focused tests for the changed capability;
- run that slice's integration tests and the growing v0.1 acceptance scenario;
- run Ruff/mypy only on the changed slice when practical;
- do not run the entire suite, dual-Python matrix, build, wheel install, and fresh
  independent review after every small repair.

A slice is blocked only by:

- failure of its declared v0.1 behavior;
- SQLite data loss or inability to reopen;
- workspace escape or permission bypass;
- credential/raw-secret exposure;
- a regression in the growing acceptance scenario.

Other reliability, compatibility, performance, refactoring, and exhaustive edge
findings are recorded in the post-v0.1 backlog.

The release-candidate gate runs once after R5:

- full Python 3.13 suite;
- the critical v0.1 acceptance and wheel-install smoke on Python 3.12;
- Ruff and strict mypy;
- source distribution and wheel build;
- clean wheel install and public import/example execution;
- SQLite create/reopen/delete and selected safe-boundary restart tests;
- one independent whole-v0.1 review;
- documentation and version audit.

The release review blocks only Critical findings and Important findings that
violate this written v0.1 contract. General hardening proposals do not expand the
release scope.

## 13. Definition of Done

Version `0.1.0` is complete only when:

- every step of the release acceptance scenario passes through public APIs;
- SQLite is the default and a supported restart continues at a safe boundary;
- unknown in-flight external work is visible as interrupted and is not replayed
  without an application decision;
- Workflow condition/loop, automatic L0-L4, Child communication, Trace, and basic
  attribution all appear in the installed-wheel example;
- permission checks contain built-in filesystem/shell work to configured
  workspaces;
- the release-candidate gate passes;
- package metadata no longer identifies the build as a development version;
- the v0.2 backlog is retained without blocking the release.
