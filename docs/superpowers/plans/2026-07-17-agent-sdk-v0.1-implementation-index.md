# Agent SDK v0.1 Implementation Plan Index

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Deliver the approved Agent SDK `0.1.0` contract through six resumable release slices, each extending one installed-wheel acceptance scenario.

**Architecture:** Preserve the current event-sourced asyncio runtime, SQLite default store, LiteLLM-only model layer, Tool registry, and public SDK facade. Add the smallest durable contracts needed for built-in Tools, Workflow control flow, automatic Context planning, Child mailboxes, and evidence-linked attribution.

**Tech Stack:** Python 3.12-3.13, asyncio, uv, Pydantic v2, LiteLLM, aiosqlite, official MCP Python SDK, PyYAML, pytest, pytest-asyncio, Ruff, mypy, build.

## Global Constraints

- Implement only the approved contract in `docs/superpowers/specs/2026-07-17-agent-sdk-v0.1-release-design.md`.
- SQLite remains the default; `InMemoryStore` remains an explicit test/transient option.
- Use LiteLLM directly for all model and structured-output calls.
- Keep Session Ledger events immutable; Context compaction creates derived views and capsules.
- Route built-in, application, MCP, and Child-control Tools through the same schema, permission, execution, and Trace path.
- Never silently repeat unknown external work after restart.
- Child capabilities only narrow parent capabilities.
- Generated Workflow text remains a candidate until the application validates and starts it.
- Prefer compatibility adapters over broad rewrites of the existing public API.
- Follow focused RED/GREEN tests in each task. Run the full suite and packaging gates only in R5.
- Commit after each task with the message specified by that plan.
- Do not begin the next slice until the current slice's checkpoint and acceptance extension are green.

## Shared Acceptance Test Ownership

The growing release scenario lives in:

- `tests/e2e/test_v01_release.py`: one public-API scenario, extended slice by slice;
- `tests/fixtures/v01_runtime.py`: deterministic LiteLLM/MCP/application fixtures only;
- `examples/v01_reference.py`: installed-wheel example finalized in R5;
- `docs/plans/releases/v0.1.md`: human-readable release ledger created in R0.

Each slice owns one named phase in `test_v01_release.py`. A phase may call a focused
fixture helper, but it must use public SDK APIs for the behavior being accepted.
No skipped or expected-failure phase counts toward the release.

## Resumption Protocol

At the end of every task:

1. Run the task's focused command.
2. Update `docs/plans/releases/v0.1.md` with the completed checkpoint and evidence.
3. Update `.superpowers/sdd/progress.md` with current slice/task, commit, and next command.
4. Commit only the files belonging to that task.

To resume:

1. Read this index and `docs/plans/releases/v0.1.md`.
2. Read the active slice plan from its first unchecked task.
3. Confirm `git status --short` contains no unexplained changes.
4. Run the last recorded checkpoint before continuing.

## Deferred Work

The following remain explicitly post-v0.1: durable File Artifact lifecycle,
parallel/foreach/subworkflow nodes, public pause/cancel/force-delete, sync facade,
distributed recovery, fair Child scheduling, vector retrieval, statistical or
causal attribution, OpenTelemetry exporters, hosted monitor service, and broad
performance/compatibility hardening.
