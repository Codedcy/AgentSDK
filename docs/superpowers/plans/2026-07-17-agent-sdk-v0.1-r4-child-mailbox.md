# Agent SDK v0.1 R4 Child Control Tools and Mailbox Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:test-driven-development for each task, superpowers:systematic-debugging for concurrency/recovery failures, and superpowers:verification-before-completion before the R4 checkpoint.

**Goal:** Let a parent Agent spawn, message, inspect, and wait for Child Runs through ordinary Tools, with durable at-least-once messaging and enforced Tool/workspace capability intersections.

**Architecture:** Make effective Run capabilities part of the execution descriptor and select a Tool catalog per Run. Generalize the existing `SubagentService` beyond Workflow-owned children. Persist one versioned mailbox per recipient plus a consumption cursor committed with each Context View. Register four control Tools whose handlers delegate to a single `ChildCoordinator`; expose the same coordinator through a small application-facing API.

**Tech Stack:** Python 3.13, asyncio tasks/semaphore, Pydantic v2, existing RuntimeCommands/RunEngine/ToolRegistry/ContextMiddleware/StateStore, pytest, pytest-asyncio.

## Global Constraints

- Child Runs use the ordinary Agent Loop, Context, Tool, permission, recovery, and Trace paths.
- Parent-to-child and child-to-parent messages share one mailbox implementation.
- A Run may address only its direct parent or direct children in the same Session.
- Delivery is at least once; immutable message records are never overwritten.
- A Context View atomically advances the recipient mailbox cursor for messages it includes.
- Child Tool/workspace capabilities are the intersection of parent, TaskEnvelope, Child Agent, and Session capabilities.
- `None` means “inherit/no additional narrowing”; an explicit empty tuple means no capability.
- A Child cannot re-expand a Tool/workspace removed by an ancestor.
- Enforce depth, children-per-parent, total-children-per-Session, and concurrent Child limits before external work.
- `wait_child` has a bounded timeout and may return `pending`; it never blocks the Agent Loop indefinitely.
- Do not add detach/cancel propagation, budgets, fair/distributed scheduling, cross-Session messaging, or arbitrary Run-to-Run messaging.

```python
class ToolCatalog:
    def __init__(self, registered: tuple[RegisteredTool, ...]) -> None:
        self._registered = {item.spec.name: item for item in registered}

    def get(self, name: str) -> RegisteredTool:
        try:
            return self._registered[name]
        except KeyError as error:
            raise AgentSDKError(
                ErrorCode.NOT_FOUND,
                "tool not found in run capability",
                retryable=False,
            ) from error

    def list(self) -> tuple[ToolSpec, ...]:
        return tuple(self._registered[name].spec for name in sorted(self._registered))

    def schemas(self) -> tuple[dict[str, Any], ...]:
        return tool_schemas(self.list())


class ToolRegistry:
    def select(self, names: Iterable[str] | None) -> ToolCatalog:
        selected_names = (
            tuple(sorted(self._registered))
            if names is None
            else tuple(sorted(set(names)))
        )
        return ToolCatalog(tuple(self.get(name) for name in selected_names))
```

Sort/deduplicate names canonically and reject unknown names before Run creation.

- [ ] **Step 3: Implement versioned mailbox snapshots**

```python
class MailboxSnapshot(BaseModel):
    model_config = ConfigDict(frozen=True, extra="forbid")

    recipient_run_id: str
    session_id: str
    version: int = Field(default=1, ge=1)
    messages: tuple[AgentMessage, ...] = ()


class MailboxCursorSnapshot(BaseModel):
    model_config = ConfigDict(frozen=True, extra="forbid")

    recipient_run_id: str
    session_id: str
    last_consumed_sequence: int = Field(default=0, ge=0)
    version: int = Field(default=1, ge=1)
```

`MailboxService.send`:

1. loads sender and recipient Run snapshots;
2. checks same Session/direct parent-child relation;
3. loads the recipient mailbox;
4. appends an immutable message with `sequence=last+1`;
5. atomically commits mailbox snapshot, `agent.message.sent`, and optional
   idempotency record with exact preconditions;
6. retries bounded precondition conflicts.

No SQL migration is needed because snapshots are generic.

- [ ] **Step 4: Consume messages with the Context View commit**

Before strategy selection, `ContextPlanner.prepare` loads messages with sequence
greater than the cursor, appends them as protected `SourceMessage`s with refs equal
to message ids, and records those ids in `ContextView.consumed_message_ids`.

The same `CommitBatch` that writes the Context View advances
`MailboxCursorSnapshot.last_consumed_sequence` to the highest included sequence.
If the commit conflicts, reload sources/cursor and rebuild. This ensures a created
view and its consumption cursor cannot diverge.
